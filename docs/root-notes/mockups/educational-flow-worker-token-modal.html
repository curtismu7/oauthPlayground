Teaching how server-to-server OAuth flows work

Request a Worker Token

Machine-to-machine (M2M) authentication: client credentials → access token

1. Your Worker App Credentials
2. Request Token

When you click Request Token, the flow:

  1. POSTs to https://auth.pingone.com/[envId]/as/token
  2. Authenticates as the worker client (client_secret_basic or client_secret_post)
  3. Requests grant_type=client_credentials
  4. Receives an opaque or JWT access_token (depends on PingOne config)
POST /[envId]/as/token Authorization: Basic base64(client_id:client_secret) Content-Type: application/x-www-form-urlencoded grant_type=client_credentials &scope=p1:read:user p1:update:user

Use the worker token to call PingOne APIs on behalf of the worker app (not a user).

Example: Read User Details
GET /v1/environments/[envId]/users/[userId] Authorization: Bearer [access_token]
Example: Invite User
POST /v1/environments/[envId]/invitations Authorization: Bearer [access_token] Content-Type: application/json { "user": { "email": "newuser@example.com" }, "messageTemplate": "WELCOME" }
Key point: The token represents the worker app itself, not a user. Use it for admin tasks, user management, and service-to-service calls. Do NOT expose it to the frontend.

The token grants these permissions (scopes). Limit scopes to what the worker app actually needs.

  • p1:read:user Read user profiles, attributes, and account info
  • p1:update:user Modify user attributes (email, phone, custom data, etc.)
  • p1:read:device Read registered devices (phones, hardware keys)
  • p1:update:device Enroll, revoke, or update devices
  • p1:read:group Read group memberships
  • p1:create:group Create and manage groups
Security best practice: Request only the scopes your service needs. Scopes are set when the worker app is created in PingOne, so update them there — not in code.
